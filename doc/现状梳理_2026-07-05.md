## 一、执行摘要

本项目是面向严肃业余网球爱好者（NTRP 2.5–4.5）的微信小程序，以 Elo 积分引擎为核心，构建"动态网球数字名片"。技术栈采用 UniApp Vue3 + TS + Tailwind（前端）与 Vercel Serverless + Supabase + Upstash Redis（后端）。

**核心结论**：项目设计与架构层面已收敛，但**代码实施进入"双轨并行"的混乱期**——后端存在两套互不兼容的实现，前端核心数据页仍为静态原型。相较于 2026-06-13 的旧审查报告，部分高风险项（Cron、登录、比赛创建）已修复，但新暴露的架构性问题和未闭环的联调项使项目实际可发布度仍低于预期。

## 二、产品方案梳理

### 2.1 产品定位（清晰度：高）

- **核心痛点**：业余网球圈约球水平不匹配，自评 NTRP 主观且静态
- **解决方案**：问卷定级 + 真实比赛 Elo 演进 = 动态战绩背书的数字名片
- **目标用户**：严肃业余玩家（NTRP 2.5–4.5），画像"张哥，32岁 IT 从业者"
- **核心指标**：激活率 >70%、履约率 >60%、次月留存 >40%、K-factor 1.5

### 2.2 信息架构（三 Tab）

| Tab | 功能 | 状态 |
|---|---|---|
| 我的名片 | 积分、NTRP 映射、五维雷达图、战绩摘要、发起比赛 CTA | 视觉完成，数据 mock |
| 战绩大厅 | 待确认提醒、历史战绩列表 | 静态硬编码 |
| 排行榜 | 好友榜 / 同城榜 | 静态硬编码 |

### 2.3 核心机制

- **比赛状态机**：0 等候 → 1 进行中 → 2 待确认 → 3 已生效 / 4 争议废弃
- **Elo 引擎**：K 因子分级（新手 40 / 高手 16），双打用队伍平均分，零和博弈
- **防赖账**：12 小时静默生效 + 3 次驳回废弃 + 分布式锁防重复提交
- **防刷分**：同对手高频交手且胜率 >80% 时奖励减半

### 2.4 设计文档完整度

15 份文档齐全（`doc/00~09` + 实施计划 + 审查报告 + 前端架构指南 + 进度看板 + README），覆盖业务背景、产品方案、问卷设计、Elo 引擎、荣誉体系、交互原型、技术架构、执行计划、密钥配置。**设计层是项目最成熟的环节**。

## 三、代码现状梳理

### 3.1 后端：双套实现并存（核心风险）

项目存在两套并行、互不兼容的后端实现，这是旧审查报告未充分指出的最大问题。

#### 实现A：三层架构（controller-service-repository）

- **入口**：`api/api.ts`（统一路由分发）
- **文件**：`api/users/`、`api/assessment/`、`api/matches/`、`api/leaderboard/`
- **特征**：使用 `shared/database.ts`、`shared/redis.ts` 单例；`MatchesService` 类封装业务逻辑
- **路由**：`/api/matches`、`/api/matches/:id/join`、`/api/matches/:id/confirm` 等（**复数 matches**）

#### 实现B：独立 serverless 函数

- **入口**：`api/match/*.ts`（每个文件独立 handler）
- **文件**：`api/match/{create,join,start,submit-score,confirm,reject,timeout-settlement,[id],list}.ts`、`api/login.ts`、`api/evaluate.ts`、`api/users/me.ts`、`api/index.ts`
- **特征**：每个文件自行 `createClient`（Supabase/Redis 重复创建）；逻辑直接写在 handler 内
- **路由**：`/api/match/confirm`、`/api/match/submit-score` 等（**单数 match**）

#### 业务逻辑不一致清单

| 业务点 | 实现A（matches.service.ts） | 实现B（match/reject.ts） | 影响 |
|---|---|---|---|
| 驳回逻辑 | `confirmScore(confirm=false)` 直接置 status=4 废弃 | 累计 `reject_count`，满 3 次才废弃 | 行为分裂 |
| Elo 计算 | 调 `matches.dto.ts` 的 `calculateEloChange`（返回 winnerChange/loserChange） | 调 `shared/elo.ts` 的 `calculateEloChange`（返回 changeA/changeB/newScoreA/newScoreB） | 函数签名不同 |
| Redis 锁 | `redis.setnx` + `expire(5s)` | `redis.set({px:5000, nx:true})` | 实现等价但代码不一致 |
| Supabase 客户端 | 共享单例 ✅ | 每文件重复创建 ❌ | 资源浪费 |

#### 路由路径冲突

- 前端 `create.vue` 调用 `POST /api/matches/:id/start` → 走实现A，但 **`api/api.ts` 未定义 `/start` 路由** → 404
- `vercel.json` cron 指向 `/api/match/timeout-settlement` → 走实现B
- 前端 `index.vue` 调用 `POST /api/login` → Vercel 优先匹配 `api/login.ts`（实现B），但 `api/api.ts` 内也有 `handleLogin`（实现A 重复）

> **结论**：后端必须收敛为单一实现。建议保留实现A的三层架构（更规范），将实现B的业务逻辑（特别是 reject.ts 的 3 次驳回计数、shared/elo.ts 的完整算法）合并进实现A，并删除实现B的独立函数文件。

#### 关键代码问题

1. **雷达图未使用 uCharts** ❌
   - `index.vue` 第 187–244 行用 `uni.createCanvasContext` 手绘雷达图
   - `@qiun/ucharts@2.5.0` 已在 `package.json` 声明但未引入使用
   - 与设计文档《07_前端架构与UI还原指南》明确要求的 `<qiun-data-charts type="radar">` 不符

2. **首页数据未刷新** ⚠️
   - `onMounted` 仅从 `uni.getStorageSync('user_info')` 恢复，未调用 `GET /api/users/me` 拉取最新积分与雷达数据
   - `matchCount=24`、`winRate=62`、`radarData=[0.5,0.5,0.5,0.5,0.5]` 均为硬编码

3. **request 导入路径不一致** ⚠️
   - `index.vue`：`@/utils/request`
   - `create.vue`：`../../utils/request`
   - 路径别名与相对路径混用，维护成本高

4. **开赛请求 404** ❌
   - `create.vue` 第 276 行调用 `POST /api/matches/${matchId}/start`
   - `api/api.ts` 路由表内**无此路径**，`matchesController` 也未见 `startMatch` 方法
   - 实现B的 `api/match/start.ts` 路由为 `/api/match/start`（单数，无路径参数），与前端调用不匹配

### 3.3 数据库设计

`database/schema.sql` 定义 4 张表，设计合理：

- `users`：含 `wx_openid`、`current_score`、`radar_data` JSONB
- `matches`：含状态机字段、`timeout_at`、`reject_count`、`last_reject_reason`
- `match_participants`：含 `snapshot_score`（开赛快照，Elo 计算基数）
- `score_logs`：积分变动流水

**不足**：无索引定义（`wx_openid` 应建唯一索引，`matches.status` + `timeout_at` 应建复合索引供 Cron 查询）；无 RLS 策略；`match_participants` 缺唯一约束防同一用户重复加入同一比赛。

### 3.4 安全与配置

- `api/login.ts` 第 6–8 行：`supabaseUrl || ''`、`supabaseKey || ''` 兜底空字符串，环境变量缺失时仍创建客户端（静默失败）
- 《实施计划》P0 已识别"默认 `WX_APP_SECRET` 需移除"，但 `api/login.ts` 当前未显式兜底默认密钥（已改善）
- `vercel.json` 未配置 `functions` 内存/超时
- 无 `.env.example` 在 `api/` 目录（实施计划提及但未落地）

## 五、关键问题与风险（按优先级）

### P0 - 阻塞发布

1. **后端双套实现未收敛**：业务逻辑分裂（驳回逻辑、Elo 函数签名、路由路径），行为不可预测
2. **开赛请求 404**：前端 `/api/matches/:id/start` 在后端无对应路由

4. **真环境零部署**：Vercel/Supabase/Upstash/微信小程序后台均未完成验收

### P1 - 影响质量

5. **Redis 锁 5 秒偏短**：Elo 计算 + 多次 DB 写入可能超时，导致重复结算
6. **雷达图未用 uCharts**：与设计文档不符，跨端适配与动画能力缺失
7. **首页未刷新用户数据**：登录后积分/雷达图为旧值
8. **`api/login.ts` 与 `api/api.ts` 的 `handleLogin` 重复**：维护隐患

### P2 - 长期债

9. **数据库缺索引与约束**
10. **无测试用例**
11. **无 CI/CD**
12. **request 导入路径不统一**
