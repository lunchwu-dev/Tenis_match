# 项目执行与实施计划 (v1.0 MVP)

**文档目的**：基于《网球名片》的产品需求、交互原型、算法设计以及 Vercel Serverless 技术架构，制定 v1.0 (MVP) 的详细实施落地计划。为研发团队提供清晰的里程碑 (Milestones) 拆解、排期建议与验收标准。

---

## 整体排期概览 (Timeline Overview)

MVP 阶段建议按 **6 周 (30个工作日)** 的研发周期进行冲刺。划分为 5 个 Sprint (阶段)。

*   **Phase 0（第1周）**：基建与环境准备
*   **Phase 1（第2周）**：用户体系与定级闭环
*   **Phase 2（第3-4周）**：核心比赛与计分状态机
*   **Phase 3（第5周）**：排行榜与自动化延时结算
*   **Phase 4（第6周）**：内测、验收与发布

---

## Phase 0: 基础设施与环境准备 (Infrastructure Setup)
**周期**：1 周
**目标**：拉齐资源，跑通“代码提交 -> 自动部署”的基础全链路，并完成数据库建表。

### 核心任务
1. **云资源采买与配置**：
   - 注册/配置 **Vercel** 团队空间，绑定 GitHub 仓库。
   - 配置 **Supabase (PostgreSQL)** 实例，拿到连接串。
   - 配置 **Upstash Redis** 与 **QStash** 实例，提取 Token。
2. **微信生态准备**：
   - 注册微信小程序主体，获取 `AppID` 和 `AppSecret`。
   - 在微信后台配置合法请求域名（Serverless API 域名）。
   - 申请并配置订阅消息模板（“赛果确认提醒”、“积分变动提醒”）。
3. **技术脚手架搭建**：
   - **前端**：初始化 Taro/Uni-app 工程，集成 TailwindCSS 和 ECharts，封装统一的 `request` 网络请求模块，处理登录态 Token。
   - **后端**：使用 Node.js / TypeScript 初始化 API Routes，接入 Supabase SDK 与 Upstash SDK。
4. **数据库初始化**：
   - 在 Supabase 中执行 `07_技术架构设计` 中列出的 SQL DDL 建表脚本。

## Phase 1: 用户体系与初始定级闭环 (Auth & Onboarding)
**周期**：1 周
**目标**：开发并验收 “小程序登录 -> 填写20+题问卷 -> 计算出初始积分 -> 生成并展示雷达图首页” 这条完整的用户启动链路。

### 核心任务
1. **微信授权与用户入库**：
   - **FE**: 开发登录页面及微信一键登录 (`wx.login`)。
   - **BE**: `POST /api/auth/login` 接口，换取 OpenID，签发自定义 JWT / Session Token，自动插入/更新 `users` 表。
2. **定级问卷系统**：
   - **FE**: 根据 UX 原型实现沉浸式滑动答题卡片。实现答题进度的本地缓存（防意外退出）。
   - **BE**: `POST /api/assessment/submit` 接口，接收答卷 JSON，结合分流逻辑与 0-100 分映射规则计算得分，更新 `current_score` 与 `radar_data`。
3. **Tab 1 我的名片 UI**：
   - **FE**: 集成 ECharts 绘制能力五维雷达图。渲染综合得分与对标的 NTRP 荣誉等级。引入分享海报 (`wx.showShareMenu`) 逻辑。

### 验收标准
- [ ] 首次进入的新用户强制查收问卷，可完整跑通 23 题且算出合理分数落库。
- [ ] 中途杀掉微信进程，重新打开能恢复到原题号断点。
- [ ] “我的名片”主页分数大屏和雷达图能正确基于 JSON 渲染。

## Phase 2: 比赛核心状态机与 Elo 结算 (Match Engine)
*这是整个系统最重要、最复杂的核心模块，占用 2 周。*
**周期**：2 周
**目标**：跑通从建房、拉人、比赛确认比分到实时完成 Elo 并发结账的核心闭环。

### 核心任务
1. **建房与房间同步机制**：
   - **FE**: 开发 Tab 1 中“发起比赛”入口，进入“比赛准备室”。实现按微信群分享的小程序卡片。
   - **FE**: 实现短轮询监听房间加入状态，房主显示“确认开赛”按钮。
   - **BE**: `POST /api/match/create` 和 `POST /api/match/join` 接口。
2. **比赛落网与积分快照**：
   - **BE**: `POST /api/match/start` 接口，关键动作：锁定四人/两人当前的积分存入 `snapshot_score` 字段，更新房间状态为进行中。
3. **比分提交与对手确认**：
   - **FE**: 定制化开发数字键盘输入比分 UI。对手收到模版消息点击进入确认详情页。
   - **BE**: **分布式锁防护**。`POST /api/match/submit_score` 与 `POST /api/match/confirm_score` 必须接入 Upstash Redis 的 `SETNX`。
4. **Elo 引擎实现**：
   - **BE**: 封装独立的 `EloService` 类。实现：双打平均分转换、阶梯 K 因子判定（2/3/4/5/8倍率）、防刷分惩罚机制（15天同对手交战超3次）、最终分值结算。

### 验收标准
- [ ] 能稳定创建单双打房间，且通过微信卡片邀请别人落座。
- [ ] “狂点提交比分”或“两人同时点确认”均不会引发数据库出现多笔 `score_logs`。
- [ ] 线下真实验证 4 个测试账号，胜负结算公式算出的分数与预期 Excel 演算无豪厘偏差。

## Phase 3: 排行榜引擎与延时静默规则 (Leaderboard & Async)
**周期**：1 周
**目标**：完成整个系统的社交激励板块与最终的容错防护网开发。

### 核心任务
1. **Redis ZSET 实时排行榜体系**：
   - **BE**: 所有触发 `EloService` 结算的地方，同步向 Upstash Redis 发送 `ZADD` 指令更新积分排名。
   - **BE**: `GET /api/leaderboard/top` 接口实现。
   - **FE**: 开发 Tab 3 排行榜界面，重点呈现该用户的自身具体排名。
2. **系统超时静默确认机制 (12h)**： // *极重要防御机制*
   - **BE**: 配置 Upstash QStash 或 Vercel Cron。房主提交比分时发送 webhook 给队列，或写个每 15 分钟触发拉取超期数据的跑批脚本。对超期 12 小时的待确认比赛执行静默过审。
3. **战绩大厅与流水查询**：
   - **FE & BE**: 开发 Tab 2 历史战绩页，分页加载 `matches` 联合 `score_logs` 表，展示过去每场比赛的赛果详情。

### 验收标准
- [ ] 排行榜刷新延迟低于 100ms，排行榜分数和用户真实数据一致。
- [ ] 不点击提交按钮，放任比赛订单过期 12 小时后，系统判定自动成功，积分发生了变化，并且历史记录中呈现为“已结算”。

## Phase 4: 测试验收与灰度发布 (Testing & Launch)
**周期**：1 周
**目标**：完成端到端的 UAT，修复遗漏体验，合规过审微信发版。

### 核心任务
1. **内部“狗食”测试 (Dogfooding)**：
   - 将体验版发给项目组成员及外部网球圈里的 10-20 名球友，约两场真实的比赛跑全流程。
2. **异常用例 (Edge Cases) 遍历检验**：
   - 测试比分争议逻辑（连续驳回 3 次导致废单）。
   - 测试定级问卷纯选择弱智选项能否成功保底 0 分。
   - 中断网络时提交请求。
3. **微信生态合规与发版**：
   - 配置《用户隐私保护指引》（涉及收集微信昵称头像、头像信息）。
   - 上传生产环境版本，提交微信团队审核。
   - 配置正式环境的 Vercel 域名（切走测试库，挂载生产 Supabase/Redis）。

### 验收标准
- [ ] 完整扫清所有 P0/P1 阻塞级 Bug。
- [ ] 微信审核通过，正式上架 MVP 1.0 版本。
- [ ] Vercel 与前端统计 Dashboard 可清晰观测首批用户的增长漏斗数据。
