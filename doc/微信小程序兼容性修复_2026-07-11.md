# 微信小程序平台兼容性审计与修复（2026-07-11）

## 目标
确保前端（UniApp Vue3 + TS）能在**微信小程序**平台上运行，并尽量还原 ARDot 设计稿的暗黑荧光风。

## 结论
✅ **项目可以正常编译为微信小程序包**（`pnpm --filter client run build:mp-weixin` → `client/dist/build/mp-weixin`，用微信开发者工具导入即可运行）。
✅ 类型检查零错误。
✅ 修复了 1 个会导致整个项目无法编译的致命错误 + 3 个微信端专属的视觉/健壮性问题。

## 预先排查的高风险点（结果：均安全）
| 检查项 | 结果 |
|---|---|
| `v-html` / `innerHTML` | 无（小程序不支持，已规避） |
| `backdrop-filter` 毛玻璃 | 无使用（注：即使使用也会被静默忽略） |
| DOM API（`document`/`window`/`querySelector`） | 无 |

| Canvas | 使用 `canvas-id` + `createCanvasContext`（兼容写法） |

## 已修复的问题

### 1. P0 — 首页模板损坏导致整个项目无法编译
`pages/index/index.vue` 第 121 行多了一个孤立的 `</view>`（早期雷达图重构的残留），编译报错 `Element is missing end tag`。移除后 `build:mp-weixin` 通过。
**影响**：不修则整个小程序构建失败、无法运行。

### 2. P1 — 雷达图在微信端可能不绘制
原 `drawRadar` 使用 `uni.createSelectorQuery().in(getCurrentInstance())`，在微信端 `rect` 可能为 `null` 导致雷达图静默消失。
改为页面级查询 `uni.createSelectorQuery()` 并加 **兜底尺寸 176px**，保证雷达图始终渲染。同步移除不再使用的 `getCurrentInstance` 导入。

### 3. P1 — TabBar 图标安卓端豆腐块
原 `⊞` / `☰` 为符号字形，在多数安卓微信客户端会显示为 □。
改为通用 emoji `📇`（名片）/ `📊`（战绩）/ `🏆`（排行），跨设备稳定渲染。

### 4. P1 — 品牌字体从未加载
设计稿的 Oswald / DM Sans 在全仓库无任何加载逻辑，会静默回退系统字体，与设计稿字形不符。
在 `App.vue` 的 `onLaunch` 内以 `#ifdef MP-WEIXIN` 包裹 `uni.loadFontFace` 注入（jsdelivr fontsource 源），**失败静默回退**，不影响运行。

## 已知平台限制（非阻塞，已在 MEMORY.md 记录）
- **`backdrop-filter` 不可用**：带 `backdrop-blur-sm` 的玻璃卡会降级为半透明平卡（无模糊），仍是同色系暗卡，视觉可接受。
- **`:active` 伪类无按压反馈**：`active:scale-95` 等按压动效在微信端不触发（需 `hover-class` 才能做按压态）。
- **中文字体走系统字体**：Oswald/DM Sans 仅含拉丁字形，主要影响数字/英文（ELO、百分比）的字形；中文始终由系统字体渲染（小程序固有约束）。
- **结构性偏差（待拆分）**：设计稿中 *Create（发起比赛）* 与 *Room（赛前准备室）* 是两个独立页面，前端目前用单个 `create.vue` 兼任，与设计稿页面结构不完全一致。
