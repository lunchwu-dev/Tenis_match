# 技术架构与部署方案设计 (v1.0)

**文档目的**：确立《网球名片》小程序的整体技术架构、服务端技术选型以及基于 Vercel 生态的云原生/Serverless 部署方案，提供所需的三方服务清单与实施指引。

---

## 一、 整体技术栈选型 (Tech Stack)

结合 MVP 阶段敏捷开发的需求以及您现有的 **Vercel** 资源，我们采用 **Serverless (无服务器) 架构**。这种架构免去了服务器运维烦恼，且天然具备自动扩缩容能力。

### 1. 前端（微信小程序端）
*   **框架**：**Taro (React + TypeScript)** 或 **Uni-app (Vue3 + TypeScript)**。
    *   *说明*：使用现代前端框架编写，不仅开发体验好，配合 TailwindCSS 可以完美还原 UX 原型，且方便后续扩展。
*   **组件与图表**：使用 ECharts for Weixin 绘制能力雷达图。

### 2. 后端（服务端 API）
*   **运行环境**：**Vercel Serverless Functions** (Node.js / TypeScript)。
    *   *说明*：利用 Vercel 托管所有的 Backend API（如 `/api/match/submit`）。代码与 Vercel 深度集成，提交 Git 即自动部署，零运维成本。
*   **框架选型**：推荐直接使用 Next.js (API Routes) 构建纯接口，或使用 Express 配合 Vercel adapter。

### 3. 数据层与外部依赖 (DB & Infrastructure)
由于 Vercel 本身只提供计算（代码运行），**您还需要准备以下云服务资源（均有良好的免费额度/Free Tier 供前期使用）**：

| 资源组件 | 推荐服务商 (Vercel 生态友好) | 核心用途 |
| :--- | :--- | :--- |
| **关系型数据库 (RDBMS)** | **Supabase (PostgreSQL)**<br>或 **Vercel Postgres (Neon)** | 存储用户资料、比赛房间、赛果流水等核心持久化数据。*推荐 Supabase，后续需要 WebSocket 时自带 Realtime 功能。* |
| **缓存与锁 (Cache/Redis)** | **Upstash Redis** | **极其关键**！用于：<br>1. 分布式锁（防高发作弊、防止同一场比赛多次算分）<br>2. 主页排行榜 (Redis ZSET)<br>3. 用户 Session 控制。 |
| **延时队列 / 定时任务** | **Upstash QStash** 或 **Vercel Cron** | 解决 [12小时赛果未被确认自动生效] 的核心痛点。 |
| **微信生态账号** | 微信公众平台 | 注册微信小程序实体，获取 `AppID` / `AppSecret`，配置业务域名、消息订阅模板。 |

## 二、 核心业务难点的 Serverless 解决路径

在传统的长驻留服务器 (如云服务器 ECS + Java/Spring) 上容易实现的功能，在 Vercel 无状态的 Serverless 环境中需要特殊设计：

### 1. 防作弊与超卖机制 (分布式锁)
*   **场景**：用户在网络不好的球场狂点“提交比分”，产生多次并发请求。
*   **方案**：在 Vercel Function 收到请求的第一时间，通过 **Upstash Redis** 执行 `SETNX (SET if Not eXists)`。
    *   例如：`SETNX match_lock_{match_id} 1 EX 5`。若获取锁失败，立刻返回“处理中请勿重复点击”。若成功，则深入 Postgres 进行状态校验与更新。

### 2. 12 小时超时自动结算积分 (延时任务)
*   **场景**：防死局机制，房主提交比分 12 小时后对手未操作，系统需自动运行 Elo 结算。
*   **Vercel 方案 A (主动触发 - 推荐)**：引入 **Upstash QStash** (专为 Serverless 设计的消息队列)。在房主提交比分时，往 QStash 发送一条延时 12 小时的消息。12小时后，QStash 会自动调用你的 Vercel API `/api/webhook/match_timeout` 来结算这局比赛。
*   **Vercel 方案 B (被动轮询)**：利用 Vercel 内置的 **Cron Jobs** 功能，写一个 `/api/cron/check_matches` 接口，配置 `vercel.json` 每 5 分钟执行一次，去 Postgres 查询所有超时的处于 `待确认` 状态的订单，批量执行结算。

### 3. 实时状态同步 (建房与等待对手)
*   **场景**：建房后，其他人扫码加入，房主界面需要立刻显示人员头像。
*   **挑战**：Vercel Serverless 并不适合维持长连接 (WebSocket)。
*   **方案**：
    *   针对 MVP 版本：小程序端采用**短轮询** (Short Polling)，在“比赛准备室”每 3 秒调用接口检查房间人员名单。
    *   进阶版本：使用 **Supabase Realtime** 订阅数据库变更。小程序直连 Supabase，当 `match_participants` 表有数据插入时，实时触发前端 UI 更新头像，且不消耗 Vercel 的计算资源。

### 4. 高性能同城/全国排行榜
*   **场景**：每次打开 Tab 3 排行榜，都要立刻看到自己及圈内的名次。
*   **方案**：使用 **Upstash Redis 的 ZSET (有序集合)**。
    *   比赛结算完，立刻向 Redis 执行：`ZADD global_leaderboard {Elo积分} {User_ID}`。
    *   查询排名极快：使用 `ZREVRANGE` 就能毫秒级拉取排名前 50 的用户，通过 `ZREVRANK` 获取属于当前用户的精确排名。

## 三、 数据模型概览 (PostgreSQL 适用版)

针对 PostgreSQL，核心数据表的结构与关联关系如下：

```sql
-- 用户表
CREATE TABLE users (
    id UUID PRIMARY KEY DEFAULT uuid_generate_v4(),
    wx_openid VARCHAR(100) UNIQUE NOT NULL,
    nickname VARCHAR(50),
    avatar_url TEXT,
    gender SMALLINT,
    current_score NUMERIC(5, 2) DEFAULT 0, -- 允许保留两位小数的 Elo 分数
    radar_data JSONB, -- 存底线、发球等雷达图数据
    created_at TIMESTAMP WITH TIME ZONE DEFAULT NOW()
);

-- 比赛房间表
CREATE TABLE matches (
    id UUID PRIMARY KEY DEFAULT uuid_generate_v4(),
    creator_id UUID REFERENCES users(id),
    match_type SMALLINT CHECK (match_type IN (1, 2)), -- 1: 单打, 2: 双打
    status SMALLINT DEFAULT 0, -- 0等候, 1进行中, 2待确认, 3已生效, 4争议废弃
    score_a SMALLINT,
    score_b SMALLINT,
    submitter_id UUID REFERENCES users(id),
    timeout_at TIMESTAMP WITH TIME ZONE, -- 12小时后的自动结算时间
    created_at TIMESTAMP WITH TIME ZONE DEFAULT NOW()
);

-- 比赛参与明细与快照表
CREATE TABLE match_participants (
    id UUID PRIMARY KEY DEFAULT uuid_generate_v4(),
    match_id UUID REFERENCES matches(id),
    user_id UUID REFERENCES users(id),
    team VARCHAR(10), -- 'A' 或 'B'
    snapshot_score NUMERIC(5, 2), -- 极其重要：开赛落座瞬间的积分快照，用于 Elo 计算基数
    confirm_status SMALLINT DEFAULT 0 -- 0未操作, 1确认, 2驳回
);

-- 积分变动流水流水表
CREATE TABLE score_logs (
    id UUID PRIMARY KEY DEFAULT uuid_generate_v4(),
    user_id UUID REFERENCES users(id),
    match_id UUID REFERENCES matches(id), -- 关联导致变动的赛事
    change_amount NUMERIC(5, 2), -- 变动值 (如 +2.5 或 -1.8)
    after_score NUMERIC(5, 2), -- 变更后的总分
    created_at TIMESTAMP WITH TIME ZONE DEFAULT NOW()
);
```

## 四、 部署清单与实施第一步

针对目前的资源环境，要让开发团队直接动手，您（或团队技术负责人）需要准备以下账号与配置：

1. **Vercel 配置**：一个可用的 Vercel 账号，关联至存放后端代码的 GitHub 仓库。
2. **Supabase 准备**：在 [Supabase](https://supabase.com) 注册建站，建立一个 Postgres 数据库实例，提取出 `DATABASE_URL` (Connection String)。
3. **Upstash 准备**：在 [Upstash](https://upstash.com) 创立一个 Redis 实例（用于锁和排行榜），提取 `UPSTASH_REDIS_REST_URL` 及其 Token。若使用延时队列，同样注册 QStash 服务提取 Token。
4. **微信小程序后台**：以公司/个人身份申请微信小程序，获取 `AppID`，并在小程序后台将上述 Vercel 提供的域名配置到 **“服务器白名单 (request 合法域名)”** 中。

将上述依赖在 Vercel 面板中配置为环境变量 (Environment Variables) 即可驱动整个项目的运转。
